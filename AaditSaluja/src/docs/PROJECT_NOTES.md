## 2026-05-06 Predictive Cold-Packing Prototype

Implemented a third storage policy:

- `predictive_cold_segments` in `src/cephfs_metadata/storage.py`.
- Plugin wrapper: `src/storage_plugins/predictive_cold_segments.py`.
- Profile runner: `src/scripts/run_cloudlab_predictor_profile.py`.

Design:

- This is no longer oracle-guided.
- New files are packed first, without inspecting `hot*` or `cold*` labels.
- The predictor observes `stat` and/or `read` events and promotes a path back
  into native CephFS after a configurable threshold.
- Evaluation metrics still count how many promoted paths were benchmark-hot or
  benchmark-cold; those labels are used only after the decision for analysis.

Profiled strategies:

- `pred_read1`: promote after one read.
- `pred_statread2`: promote after two stat/read events.
- `pred_statread4`: promote after four stat/read events.

CloudLab result directories:

- `report/results/cloudlab-predictor-profile-20260506/`
- `report/results/cloudlab-predictor-profile-native-20260506/`

Same-window native baselines:

- `cold70_access20`: `360.23` ops/s.
- `cold90_access10`: `410.46` ops/s.
- `cold90_access20`: `383.04` ops/s.

Best predictor result by scenario:

- `cold70_access20`: `pred_statread4`, `2608.79` ops/s, `7.24x` native.
- `cold90_access10`: `pred_read1`, `5714.55` ops/s, `13.92x` native.
- `cold90_access20`: `pred_read1`, `5492.51` ops/s, `14.34x` native.

Best-behaved strategy:

- `pred_statread4` is currently the cleanest paper candidate because it avoids
  most cold false promotions:
  - `cold70_access20`: 909.0 hot promotions, 114.5 cold promotions.
  - `cold90_access10`: 309.0 hot promotions, 6.5 cold promotions.
  - `cold90_access20`: 309.5 hot promotions, 74.0 cold promotions.

Interpretation caveat:

- The old predictor profile is not paper-reportable. The investigation found
  two benchmark artifacts: packed `stat()` was a pure in-memory index lookup,
  and hot churn creates were packed until individual file paths were promoted.
  That made the predictor pay much less CephFS metadata cost than oracle/native
  on important hot phases.
- Local fixes: `packed_stat_mode=index`, predictor directory promotion, and
  seeded-random hot-churn parent selection.
- A small serial post-fix smoke on the rebuilt allocation (`cold90_access10`,
  `file_count=300`, `ops=900`, seed `92`) produced native `3337.39` ops/s,
  oracle `4668.30` ops/s, and fair predictor `1284.03` ops/s. This is only a
  smoke, but it confirms the earlier predictor-over-oracle result was an
  artifact.
- For paper claims, frame the predictor as a new design point only after a full
  six-scenario rerun with fair options, at least three repeats, and error bars.

Representative workload references to use for validation:

- IOR/mdtest for filesystem metadata create/stat/remove rates.
- Filebench fileserver/webserver/mailserver-style workloads, scaled to avoid
  tiny default working sets.
- YCSB-style `uniform`, `zipfian`, and `hotspot` request distributions for
  skewed access generation.
- SNIA IOTTA traces for trace-shaped locality checks if time permits.

## Goal

Build and evaluate metadata-management techniques for CephFS, especially for metadata-heavy small-file workloads.

The proposal has two main implementation tracks:

1. Predictive metadata placement: track recent per-directory metadata activity, identify emerging hot directories, and proactively pin or migrate subtrees before CephFS's reactive balancer becomes a bottleneck.
2. Small-file packing layer: reduce metadata pressure by storing many tiny logical files inside larger append-only segment files, with a separate lookup index.

The project should first demonstrate CephFS metadata bottlenecks clearly, then compare proposed changes against native CephFS and static subtree pinning.

## Proposal Summary

Problem: CephFS is useful for large shared storage, but metadata-heavy workloads remain painful. CephFS distributes directory subtrees across MDS daemons using dynamic subtree partitioning, but this is mostly reactive. Tiny-file workloads can overload metadata paths even when payload I/O is modest.

Planned baselines:

- Native CephFS default balancer.
- CephFS with static subtree pinning.

Planned workloads:

- `mdtest`.
- Custom create/stat/read/delete scripts over large numbers of small files.
- Mixed workloads to check whether metadata improvements hurt normal I/O.

Planned metrics:

- Metadata throughput.
- p95/p99 latency for create/open/stat/delete/read where available.
- MDS CPU and load distribution.
- Metadata-pool write volume versus data-pool write volume.
- Lookup overhead and compaction overhead for the small-file overlay.

Goal tiers:

- 75%: Set up CephFS, reproduce a metadata bottleneck with `mdtest`, implement a working predictive policy.
- 100%: Implement both predictive placement and small-file packing, compare against native CephFS, produce clear throughput, tail-latency, and MDS-load graphs.
- 125%: Train a small neural network to predict emerging metadata hotspots from recent access patterns.

## Midway Report Summary

Midway work focused on literature review and preliminary CephFS bottleneck experiments.

Literature reviewed:

- Mantle: programmable metadata placement in CephFS; custom placement can outperform default balancer.
- Lunule+ and related work: CephFS subtree partitioning is reactive and migration-heavy.
- IndexFS/Giga+: hash-based distribution can balance load but sacrifices locality. A hash-based distribution baseline may be useful.
- TableFS: stores metadata and tiny files in a key-value store using an LSM-tree to reduce metadata overhead.
- DeltaFS: log-structured ingest and compaction, close to the proposed packing layer.
- HDFS HAR/AHAR: simpler file grouping; useful tradeoff between reduced metadata pressure and lookup overhead.
- HopsFS/HopsFS++ still needs review, especially colocating small-file data with metadata.

Reported CloudLab setup:

- 4 nodes total.
- 1 client-only benchmark driver.
- 3 storage/metadata server nodes.
- Each server node had mon, mds, and osd.
- MDS setup: 2 active, 1 standby.
- Ceph also added a monitor to the client node by default; this should be removed for consistency in final runs.

Preliminary workload dimensions:

- File count varied while holding total data size fixed at 64 MiB.
- Random versus sequential access.
- Read-heavy, write-heavy, and balanced workloads.
- Each workload performed create/write, stat, read, and delete.

Preliminary result:

- Metadata-pool writes increased clearly as file count increased while total payload stayed fixed.
- Higher file counts increased metadata-style work and variation across workload mixes.

Important caveat: the 64 MiB total data size was only for quick prototyping. Future runs should use larger datasets to avoid CPU-cache and inline-file artifacts.

- Pinning diagnosis follow-up:
  - Question tested: why do pinning strategies perform worse than default even
    on skewed hot-directory workloads?

      or cross-rank forwarding.

      directory. For short create-heavy workloads, forcing a pin can be worse
      than leaving that authority in place.

## Next Technical Steps

Progress on oracle-policy infrastructure:

- Added a storage-level policy backend hook so policy callbacks no longer have
  to talk directly to the raw filesystem backend.
- `oracle_cold_segments` now defaults to a logical policy directory view. This
  lets existing policy plugins see the same hot/cold workload namespace that
  native storage sees, even when cold directories are virtualized.
- For virtual cold directories, policy pin requests are translated onto the
  physical packed namespace. The default target is the packed store directory,
  with storage options for `store`, `segments`, or `root`.
- Storage metrics now report `policy_dir_view`, `cold_policy_pin_target`,
  `policy_pin_translations`, and `policy_pin_deduplications` so CloudLab runs can
  show whether the oracle path actually exercised this compatibility layer.

Progress on bridging native's write-path advantage:

- Added cold data batching to `oracle_cold_segments`.
- Before this, the oracle packed path batched the binary index journal but still
  issued one small physical segment write per packed cold file.
- New storage option: `cold_data_batch_bytes`, default `1048576`.
- Packed cold writes now buffer contiguous segment data and flush on threshold,
  read, or phase sync. This gives the packed path a native-like advantage:
  fewer small physical writes are visible to CephFS.
- New metrics: `cold_data_batch_bytes`, `data_flushes`,
  `buffered_data_bytes`, `total_packed_cold_writes`, and
  `packed_cold_bytes`.
- New derived metrics: `logical_creates_per_data_flush` and
  `packed_cold_bytes_per_data_flush`.
- Local smoke result with 64 files, 128-byte payloads, and a 1024-byte cold data
  batch:
  - Batched path: `7` data flushes.
  - Unbatched path: `56` data flushes.
  - This confirms the mechanism reduces physical write calls; CloudLab is still
    needed to measure end-to-end performance impact.

Progress on embedding the packed path deeper into CephFS:

- Added a generic backend `set_xattr()` hook for Python 3 POSIX/libcephfs runs.
- Added generic backend object I/O hooks:
  - `write_object(pool, object_id, data, offset)`.
  - `read_object(pool, object_id, offset, size)`.
  - `remove_object(pool, object_id)`.
- Added optional CephFS file-layout xattr support to `oracle_cold_segments`.
- Added `src/scripts/run_cephfs_py3_bench.sh` so the Python 3 benchmark runner and
  storage plugins can be exercised through container `libcephfs` when the local
  CephFS stack is healthy. The older `run_cephfs_bench.sh` still uses the
  Python 2 benchmark path and cannot test these storage plugins.
- New storage options:
  - `cold_data_backend=cephfs|rados|auto`, default `cephfs`.
  - `rados_pool=<pool>`, default `cephfs_data`.
  - `rados_namespace=<prefix>`, default `cs2640-packed`.
  - `ceph_layout_scope=off|store|segments|all`, default `segments`.
  - `ceph_stripe_unit=<bytes>`.
  - `ceph_stripe_count=<count>`.
  - `ceph_object_size=<bytes>`.
- The packed layer applies these layout xattrs to the packed segment directory
  and newly-created segment files, depending on scope. This lets segment files
  use CephFS's native layout machinery instead of behaving only like opaque
  byte-addressed files.
- With `cold_data_backend=rados`, packed cold payload bytes bypass CephFS segment
  files and are written as lower-layer objects. The CephFS namespace still stores
  the packed index and hot files. This is closer to a native Ceph design because
  cold data no longer consumes CephFS file inodes or segment-file metadata.
- New metrics:
  - `cold_data_backend_requested`.
  - `cold_data_backend`.
  - `rados_pool`.
  - `rados_namespace`.
  - `rados_objects_created`.
  - `ceph_layout_scope`.
  - `ceph_layout`.
  - `layout_xattrs_attempted`.
  - `layout_xattrs_applied`.
  - `layout_xattr_failures`.
