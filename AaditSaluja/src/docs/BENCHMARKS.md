# Benchmark Plan

Last updated: 2026-05-06

This project uses a small in-repo benchmark runner for local development and
repeatable experiment orchestration. It is not meant to replace established
benchmarks in the final report. Instead, it mirrors the workload shapes we need
now and leaves room to add external `mdtest`, IOR, and Filebench runs on the
later Linux/CloudLab setup.

## Benchmarks and Baselines

Primary baselines:

- Native CephFS with its default metadata behavior.
- Native CephFS with explicit static subtree pins using `ceph.dir.pin`.
- Native CephFS with a simple predictive hot-directory pinning policy.
- Native CephFS through a minimal append-only small-file packing layer.

Primary workload families:

- `mdtest_tree`: an `mdtest`-style directory tree benchmark with create, stat,
  read, and delete phases. `mdtest` is the standard metadata benchmark bundled
  with IOR and is widely used for parallel file-system metadata testing.
- `sprite_lfs_smallfile`: a classic small-file create/read/delete shape based on
  the Sprite LFS small-file benchmark: many 1 KiB files, then read, then delete.
- `filebench_varmail_like`: a Filebench-inspired mail workload with small-file
  create, read/stat, and delete operations. This gives a macro-style workload
  that is less phase-separated than `mdtest`.
- `hotdirs_zipf`: a skewed hot-directory workload for predictive placement. Most
  operations target one directory, with a tail across other directories.
- `ycsb_file_skew`: a YCSB-style file workload with a measured load/create
  phase, read phase, and update phase. It supports file-level `zipfian` and
  `hotspot` distributions, a configurable update fraction, a configurable
  generator-hot file fraction, and predictor evaluation metrics for false
  hot-directory classification.
- `predictor_false_hot_churn`: a follow-up downside benchmark for predictor
  precision. It creates probe files in generator-cold directories, repeatedly
  reads them to make the directory-hotset predictor falsely mark those cold
  directories hot, then creates a new wave of cold files in those same
  directories. The expected comparison is predictor versus oracle and
  `predictor_nolearn`; a throughput drop shows that false-hot classification
  can destroy packing benefit for future creates.
- `oracle_hotcold_mix`: an oracle-guided mixed workload that creates a large
  cold bulk set, a smaller known-hot set, then drives repeated hot-path
  metadata operations plus hot-directory churn. It now accepts
  `--oracle-cold-fraction` to control how much of the dataset is cold and
  `--oracle-cold-access-fraction` to control how often access phases touch cold
  files instead of the hot set.

## Local Runner

Run the POSIX backend on any local path:

```sh
./src/scripts/run_posix_bench.sh --suite quick
```

Run the local sanity simulator against the same Python storage implementations:

```sh
./src/scripts/simulate_local_vs_native.py \
  --file-count 512 \
  --ops 2048 \
  --dirs 32 \
  --workers 8
```

This uses `--backend simulated-ceph`, which still performs real local POSIX
file operations for correctness but injects configurable Ceph-like operation
costs for metadata operations, file creates, object reads/writes, xattrs, and
payload transfer. It is useful when the 4-node cluster is unavailable because it
compares native, `oracle_cold_segments` on CephFS segment files, and
`oracle_cold_segments` on RADOS-style cold objects under one fixed cost model.
It is not evidence of absolute CephFS performance.

Run the local Docker/CephFS backend through `libcephfs` inside the monitor
container:

```sh
./src/scripts/cephfs_up.sh
./src/scripts/run_cephfs_bench.sh --suite quick
```

Run the Python 3 storage-plugin path through container `libcephfs`:

```sh
./src/scripts/cephfs_up.sh
./src/scripts/run_cephfs_py3_bench.sh \
  --suite custom \
  --workload oracle_hotcold_mix \
  --storage oracle_cold_segments \
  --storage-opt virtual_cold_dirs=true \
  --storage-opt cold_data_batch_bytes=1048576
```

Outputs are written as JSON and CSV under `results/` by default. The JSON file
contains:

- run metadata: host, backend, root, timestamps, arguments.
- phase rows: operation count, elapsed time, throughput, mean/p50/p95/p99
  operation latency.
- run-level measured totals: `measured_seconds`, `measured_operations`, and
  `measured_ops_per_sec`.
- derived structural metrics under `derived_metrics`, such as estimated physical
  files created, materialized directories, packed-create fraction, and index
  bytes per packed record when the storage plugin exposes enough information.
- Ceph snapshots from `ceph df` and `ceph fs status` when available.

Measurement note:

- `elapsed_seconds` is whole-run wall clock.
- `measured_seconds` is the sum of timed benchmark phases.
- Storage plugins can buffer metadata/index work, so the runner now calls
  `storage.sync()` before stopping each phase timer. This keeps deferred journal
  writes inside the phase that caused them.

The container image exposes `cephfs` only through Python 2, so
`src/scripts/run_cephfs_bench.sh` uses a small Python-2-compatible runner. The main
runner in `src/cephfs_metadata/benchmark_runner.py` is the preferred path for
POSIX-mounted CephFS on Linux.

## Placement Policies

The Python 3 runner resolves policy names to plugin files under `src/policies/`.
Use `--policy <name>` for `src/policies/<name>.py`, or `--policy-file <path>` for an
explicit file.

- `none`: native CephFS behavior. This is the default baseline.
- `static`: explicit static subtree pinning. After benchmark directories are
  created, the policy sets `ceph.dir.pin` round-robin across `--pin-ranks`.
- `predictive`: a simple sliding-window hot-directory tracker. It observes the
  parent directory of each metadata operation and pins a directory once it has at
  least `--hot-threshold` operations inside the last `--hot-window` operations.
- `LLM_policy`: a separate heuristic policy file. It does not call an LLM at
  runtime; it encodes a candidate policy that combines hotness, parent fanout,
  and cooldown to avoid over-pinning noisy directories.

Policy decisions are stored in each JSON result under `policy_events`. CSV result
rows include the active policy name.

Static baseline:

```sh
./src/scripts/run_cephfs_bench.sh \
  --suite custom \
  --workload hotdirs_zipf \
  --file-count 10000 \
  --file-size 512 \
  --dirs 64 \
  --policy static \
  --pin-ranks 0,1
```

Predictive baseline:

```sh
./src/scripts/run_cephfs_bench.sh \
  --suite custom \
  --workload hotdirs_zipf \
  --file-count 10000 \
  --file-size 512 \
  --dirs 64 \
  --policy predictive \
  --pin-ranks 0,1 \
  --hot-window 256 \
  --hot-threshold 96
```

Explicit policy file:

```sh
./src/scripts/run_posix_bench.sh \
  --suite custom \
  --workload hotdirs_zipf \
  --root /mnt/cephfs/bench \
  --policy-file src/policies/example_predictive.py \
  --pin-ranks 0,1 \
  --policy-opt hot_window=256 \
  --policy-opt hot_threshold=96
```

Policy plugin files expose `create_policy(config)` and return an object with
`on_dirs_created(backend, dirs)`, `before_operation(backend, operation, path)`,
and `events()`. See `src/policies/README.md` and `src/policies/example_predictive.py`.

LLM-style heuristic policy:

```sh
./src/scripts/run_posix_bench.sh \
  --suite custom \
  --workload hotdirs_zipf \
  --root /mnt/cephfs/bench \
  --policy LLM_policy \
  --pin-ranks 0,1 \
  --policy-opt hot_threshold=96 \
  --policy-opt window=256 \
  --policy-opt fanout_threshold=8
```

## Small-File Packing Layer

The Python 3 runner supports storage plugins with `--storage <name>` or
`--storage-file <path>`. The minimal packing layer is
`src/storage_plugins/append_segments.py`.

Native file layout:

```sh
./src/scripts/run_posix_bench.sh \
  --suite custom \
  --workload sprite_lfs_smallfile \
  --root /mnt/cephfs/bench \
  --storage native
```

Packed logical-file layout:

```sh
./src/scripts/run_posix_bench.sh \
  --suite custom \
  --workload sprite_lfs_smallfile \
  --root /mnt/cephfs/bench \
  --storage append_segments \
  --segment-size 67108864
```

The packing layer stores logical file payloads in append-only segment files under
`<workload-root>/__packed/segments/` and records logical path mappings in a
separate append-only JSON-lines index at `<workload-root>/__packed/index.log`.
The JSON result includes `storage_metrics`, including segment count, index-log
bytes, logical records, live records, live bytes, and tombstones.

Oracle-guided hybrid packing:

```sh
./src/scripts/run_posix_bench.sh \
  --suite custom \
  --workload oracle_hotcold_mix \
  --root /mnt/cephfs/bench \
  --storage oracle_cold_segments \
  --storage-opt hot_prefixes=hot \
  --segment-size 67108864 \
  --file-count 20000 \
  --file-size 4096 \
  --dirs 128 \
  --ops 40000
```

Hot/cold sensitivity run:

```sh
./src/scripts/run_posix_bench.sh \
  --suite custom \
  --workload oracle_hotcold_mix \
  --root /mnt/cephfs/bench \
  --file-count 3000 \
  --file-size 4096 \
  --dirs 64 \
  --ops 12000 \
  --workers 8 \
  --oracle-cold-fraction 0.70 \
  --oracle-cold-access-fraction 0.10 \
  --storage oracle_cold_segments \
  --storage-opt hot_prefixes=hot \
  --storage-opt virtual_cold_dirs=true \
  --storage-opt index_batch_bytes=1048576 \
  --storage-opt index_batch_records=512 \
  --storage-opt cold_data_batch_bytes=1048576 \
  --storage-opt cold_data_backend=cephfs
```

This storage leaves files under `hot*` directories as native CephFS files so
they can be pinned or served directly, while packing only the `cold*` side into
segments with a compact batched binary journal.

Predictive cold packing:

```sh
./src/scripts/run_posix_bench.sh \
  --suite custom \
  --workload oracle_hotcold_mix \
  --root /mnt/cephfs/bench \
  --file-count 3000 \
  --file-size 4096 \
  --dirs 64 \
  --ops 12000 \
  --workers 8 \
  --oracle-cold-fraction 0.90 \
  --oracle-cold-access-fraction 0.10 \
  --storage predictive_cold_segments \
  --storage-opt virtual_cold_dirs=true \
  --storage-opt index_batch_bytes=1048576 \
  --storage-opt index_batch_records=512 \
  --storage-opt cold_data_batch_bytes=1048576 \
  --storage-opt cold_data_backend=cephfs \
  --storage-opt promotion_threshold=4 \
  --storage-opt promotion_triggers=stat,read
```

`predictive_cold_segments` does not use `hot*` or `cold*` labels when deciding
where to place new files. It packs new files first, observes stat/read traffic,
and promotes paths back into native files after `promotion_threshold` matching
events. The benchmark still records label-based evaluation metrics, such as
hot versus cold promotions, but those labels are only for analysis.

For fair predictor comparisons, include these options:

```sh
--storage-opt packed_stat_mode=index \
--storage-opt predictor_directory_promotion=true
```

`packed_stat_mode=index` prevents packed logical stats from being pure in-memory
Python dictionary lookups by touching the packed index file. Directory promotion
causes later creates under a promoted hot directory to use native CephFS, which
keeps hot churn comparable with the oracle/native hot path.

Current best predictor smoke candidate:

```sh
--storage predictive_cold_segments \
--storage-opt predictor_strategy=directory_hotset \
--storage-opt predictor_directory_promotion=true \
--storage-opt predictor_promote_existing=false \
--storage-opt predictor_dir_event_threshold=8 \
--storage-opt predictor_dir_distinct_threshold=3 \
--storage-opt promotion_threshold=1 \
--storage-opt promotion_triggers=read \
--storage-opt packed_stat_mode=index
```

This mode learns hot parent directories from repeated read traffic, makes future
creates under those directories native, and leaves existing packed files packed.
That avoids making read/stat operations pay synchronous rewrite/promotion cost.
On the 2026-05-06 `cold90_access10` smoke with 800 files and 2400 ops, it
reached `4143.12` ops/s versus oracle `4593.31` and native `727.46`.

Policy integration for oracle storage:

- By default, `oracle_cold_segments` exposes the original logical workload
  directory list to policy plugins via `policy_dir_view=logical`.
- Hot directories are real CephFS directories, so policy pins target those
  directories directly.
- Cold directories can remain virtual. If a policy tries to pin one, the storage
  layer translates that logical pin to the packed physical namespace. The default
  target is `cold_policy_pin_target=store`, which pins
  `<workload-root>/__cold_packed`; alternatives are `segments` and `root`.
- For the older behavior, use `--storage-opt policy_dir_view=physical`.
- Cold data writes are batched with `cold_data_batch_bytes`. This reduces the
  number of small physical `write_at()` calls into segment files; result JSONs
  report `data_flushes`, `buffered_data_bytes`, and derived data-flush ratios.
- Packed segment files can optionally use CephFS file-layout xattrs:
  `ceph_stripe_unit`, `ceph_stripe_count`, and `ceph_object_size`. Use
  `ceph_layout_scope=segments` to apply these to the packed segment directory
  and new segment files. Result JSONs report layout xattr attempts, successful
  applications, and sampled failures.
- Implementation note: CephFS uses `ceph.dir.layout.*` xattrs for directory
  default layouts and `ceph.file.layout.*` xattrs for individual files. The
  storage layer applies the directory form to packed segment directories and
  the file form to created segment files.
- Cold payloads can optionally use `cold_data_backend=rados`, which writes the
  packed cold segment bytes to lower-layer RADOS-style objects instead of CephFS
  segment files. On POSIX this uses a local object-store simulation for
  correctness testing; inside Ceph/libcephfs it uses the Python `rados` binding
  when available.

Cold-path correctness smoke:

```sh
./src/scripts/check_oracle_storage_correctness.py
```

This checks hot native writes plus cold packed writes, reads, logical stat, and
unlink/cleanup for both `cold_data_backend=cephfs` and `cold_data_backend=rados`
using the simulated backend.

## Fairness Notes

When comparing variants, keep these controls fixed:

- same workload
- same file count and file size
- same `--dirs`, `--ops`, `--workers`, and seed
- same backend and CephFS mount
- same cleanup mode (`--keep-data` or not)

For the 2026-05-06 hot/cold matrix, the six configs are:

| Scenario | Cold dataset fraction | Cold access fraction |
| --- | ---: | ---: |
| `cold70_access0` | 70% | 0% |
| `cold70_access10` | 70% | 10% |
| `cold70_access20` | 70% | 20% |
| `cold90_access0` | 90% | 0% |
| `cold90_access10` | 90% | 10% |
| `cold90_access20` | 90% | 20% |

The matrix runner in `src/scripts/run_cloudlab_hotcold_matrix.py` is intentionally
serial. It randomizes global run order, reuses the same seed for native and
hybrid variants within a scenario/repeat, checks Ceph health before every run,
and writes raw JSON/CSV/log files plus `summary.csv`, `phase_summary.csv`, and
`summary.md`.

Within `oracle_hotcold_mix`, access-phase stat/read paths are sampled using a
seeded RNG with replacement. For each access operation, the runner first draws
whether it should hit the cold set according to `--oracle-cold-access-fraction`,
then chooses a random file from the selected hot or cold set. Hot churn file
names are unique, and their parent hot directory is now chosen with the same
seeded RNG instead of deterministic round-robin assignment.

The predictor profile runner in `src/scripts/run_cloudlab_predictor_profile.py`
compares the oracle upper bound with non-oracle predictive promotion strategies.
It now includes `packed_stat_mode=index` in its base storage options. Use it for
quick strategy sweeps before committing to a full six-scenario, three-repeat
matrix.

Recommended external-validation workload ladder:

- Run `mdtest` from IOR for direct filesystem metadata create/stat/remove rates
  under controlled directory trees.
- Run scaled Filebench macro workloads such as fileserver, webserver, and
  mail/varmail-style personalities after resizing datasets to exceed RAM/cache
  artifacts.
- Add YCSB-style `uniform`, `zipfian`, and `hotspot` access generators to the
  in-repo workload so predictor decisions are tested against standard skew
  models, not only the hand-written hot/cold split.
- If time permits, replay or synthesize from SNIA IOTTA traces to check whether
  predictor promotion behavior survives trace-shaped locality.

Residual risk: the runner does not remount CephFS or drop OS/Ceph caches between
runs. It avoids same-path cache reuse by giving each run a unique workload root,
and randomized order plus repeated trials reduces cache/order bias. For a
publication-grade run, add explicit cache-dropping or reboot/remount cycles and
record their cost.

## Caveats

- The local Docker setup is single-node, single-MDS, amd64 emulated on Apple
  Silicon. Use it for correctness and instrumentation only.
- The local container cannot run POSIX mounted benchmarks because FUSE is not
  available. The `libcephfs` backend is the local workaround.
- The local container runner supports the built-in policies, but Python 3 policy
  and storage plugins are intended for the POSIX/Linux path.
- `ceph df` pool occupancy can return to its starting value after cleanup. For
  final write-volume graphs, collect cumulative OSD/perf counters or time-series
  pool I/O samples during the run on the Linux cluster.
- Large final runs should use working sets that exceed CPU cache effects and
  avoid inline-file artifacts. The `standard` suite is still modest; final
  experiments should scale counts and clients significantly.
