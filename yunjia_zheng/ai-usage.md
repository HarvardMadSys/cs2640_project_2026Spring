# Lesson learnt when using agent

The shared partition results in this report partly came out of an exploration in which I used a coding agent as the primary collaborator. This section traces how I used the agent at each stage, what worked, and where it stumbled.

## Coding with agent

The agent's first role was code navigation. I used Claude Code to trace the write path and the query path through the Milvus source, asking it to identify the relevant files, summarise the function of each component, and describe the data flow, each of which I then verified against the source. The summaries were accurate enough to support reliable navigation of a codebase I had not authored, and the cycle of reading an explanation and then checking the cited code proved an efficient way to build an internal model of the system.

The agent was equally effective for the routine engineering around the experiments. It produced the database setup scripts and most of the benchmark harness with little supervision. For the benchmark itself I worked in plan mode, outlining the intended experiment and using the resulting discussion to surface edge cases before any code was written. The notable outcome was a dataset construction algorithm that targets two properties of the workload simultaneously. First, each shared and private partition is built to a prescribed size. Second, for every query, its top-$k$ ground-truth neighbors are distributed across the shared and private partitions according to a prescribed ratio. Imposing both constraints ensures that the workload exposes the access pattern under study rather being queried implicitly in the raw embeddings. Developing this algorithm interactively was substantially faster than working it out alone, and the resulting code required only minor revisions before use.

## Research with agent

### Setup: an open-ended prompt with guardrails

I handed the diagnostic task to the agent as a single open question. I told it that per-tenant throughput collapses when both tenants are served concurrently, and asked it to design experiments that would confirm the root cause, run them, log their output to disk, inspect the results for findings inconsistent with its current hypothesis, and redesign the experiments accordingly. The agent was pointed at the source tree for reference and at a `requirements.md` that fixed its operating envelope. A handful of ground rules helped keep the loop tractable. First, the agent was free to tune the Milvus configuration file but was asked to record any knob it touched. Second, the partition and collection layout already loaded into the database was treated as immutable, so the agent could craft additional partitions but never delete an existing one. Third, the existing benchmark scripts were read-only, and any new or modified script lived under `./explore`. Fourth, every main hypothesis, intermediate analysis, and inconsistency encountered along the way was appended to a running `progress.md` log.

### What the agent chose to do

Within these constraints the agent split the investigation into three phases. The first was an offline pass over the existing benchmark CSVs, in which it aggregated per-tenant throughput and latency across the available runs and tried to build a coherent picture of where the collapse was concentrated. The second was a six-probe decomposition, in which solo and concurrent traffic was directed at the shared partition, the private partitions, or both, so that the contribution of each access pattern to the slowdown could be isolated. The third was a direct measurement of thrashing, in which kernel-side page cache counters and per-segment residency snapshots were collected to test whether the concurrent-both regime was alternating its resident set faster than the working set could stay warm.

### Iterative process and human in the loop

The first summary pass over the existing benchmark logs was fast and its conclusions looked reasonable to me. Designing new experiments was harder. The agent tended to mimic the experiments I had already written, which contrasted querying one tenant over its shared and private partitions with sixteen threads against querying both tenants with eight threads each. Its first design was a cartesian product (private only, shared only, private + shared) × (per tenant, both), which is not meaningful in this setup since no real workload queries the private partition in isolation. I read this as the agent reasoning about runtime composition rather than tenant semantics, and I was curious how long it would take to realise that a search always traverses every loaded segment, so the partition label does not matter and only the total segment count does. So I kept it running. It stayed in that loop for nearly an hour without progress, and although I had asked it to reason from the source code, it rarely opened a file. It eventually added a second experiment in which one tenant's data is warmed up first and the other tenant's queries are then issued, which moved closer to the conclusion that segments are being thrashed in and out of memory but did not quite get there. After that I gave up waiting and told it to read the segment abstraction in the source directly, but even with that pointer the experimental design did not converge after several more hours. My takeaway is that when the space of plausible reasoning paths is wide, the agent will not reliably pick the productive one unless I point it out explicitly.

The agent also rarely asked questions of its own. Reading back through `progress.md` afterwards, I noticed it had latched onto the raw numbers, treating a 70% reduction and a 60% reduction as a large gap and repeatedly asking why the two values differed, rather than stepping back to ask whether either of them was the right thing to be measuring or whether it is the main concern compared to other behavior in the system. I eventually had to interrupt and point it at the thrashing measurement directly. In hindsight my instruction to "reason about inconsistencies and redesign the experiment" should have been made explicit, namely that I wanted it to find additional measurements that corroborate or refute the current hypothesis, although even that phrasing still feels too open-ended for the agent to act on without further nudging.

## Random thoughts on agents

This is the only handcrafted paragraph in this report, without any polishing from the agent, and no auto-completion (anoying when it comes up with something I want to write about. Keep typing fast would help though).

I'm not a big fan of ai partly because I am a bit afraid of how fast the agents are improving and whether it will challeng my identity as a researcher. Not to say that they will take over in the near future, but more of the cognitive burden verifying the agent's output, not being distracted by the seemly-correct direction and insisting to have a plan in mind first. It is the real battle. On the other hand, it is much easier to take the ideas from agents for granted and miss the chance to practice how to get there. (Such moments always remind me of the saying from high school, 'Seems easy when you read the key, but get it wrong in practice [in the context of exams]') I feel especially bad if the algorithm agent proposed works perfectly, but it does not echo in my mind at all. There's no personal connect to the intuition and insights although I totally got the explanation. Maybe a more general phrasing for this scenario is that, there might be many null space in the solution space, and I believe everyone will pick the one that best matches with their life philosophy. The path approaching it is important.

Although a lot of worries and complaints above, I'm very impressed about the capability of agents. One of my favorite sci-fi novel is *Imaginary Magnitude* by Stanisław Lem. There is one story called *Golem XIV*, where some kind of highly intelligent computer is built for military purpose, but then they evolve to have their philosophical reflection (and sarcasm) on the human evolution and free will. I read it in my freshman and a similar discussion came out after the LLM became popular. One quote that I still remember is 'I or something like me will give you the fruits of this knowledge. But only the fruits, not the knowledge itself, for it will not fit into your intellects. You will become wards then, like children, except that children grow into adults, whereas you will never grow up. When a higher Intelligence presents you with something you are unable to grasp, your Intelligence eclipses it.' There are some more imaginary scenarios, such as the intelligence can come up with a masterpiece of writing because it can see there is an asymmetry in some high-dimension space where the best works in histry all share a symmetric pattern, while the human can never see. It is a very interesting read.

It's been a while not to collect my thoughts and write in this manner. I need to be so careful and alert that do not hit tab when I'm a bit tired tracing my thoughts and not to just accept the agent's completion. And without proper exercise for quite some time, a rest is uergently needed after writing at this length. So bad, but maybe it's just another style I should be used to when working with agents, and sometimes come back to the old practice for some nostalgic callbacks.
